# ADR-003: 技術スタックの転換とSVGアートプロジェクトの導入

## ステータス
提案中

## コンテキスト
現在のプロジェクトはPHP（API）とVanilla JS（Frontend）で構成されていますが、開発体験の向上と、大量のSVGを扱う高負荷な「SVGアート」作品の制作に向けて、よりモダンで高速な技術スタックへの転換を検討しています。
ユーザーの要望に基づき、PHP/Pythonからの脱却を図り、GoおよびRust/WebAssembly (Wasm) を中心とした構成へ移行します。

## 決定事項
1. **技術スタックの刷新**:
   - **Backend**: PHPに代わり、**Go** を採用。Vercel Serverless Functions (Go runtime) を活用する。
   - **Frontend (Art Engine)**: 大量のSVGを高速に計算・描画するため、**Rust + WebAssembly (Wasm)** を採用。
2. **新プロジェクト: SVG Cloud Art**:
   - プロジェクト内の数百のSVGアイコンを動的に配置し、幾何学的なパターンやアニメーションを生成する「デジタルアート」機能を `art/` ディレクトリに構築する。
3. **ディレクトリ構造**:
   - `art/`: Rustソースコード、Wasmビルド結果、およびアート用フロントエンドを配置。
   - `api/go/`: Goによる新しいAPIエンドポイントを配置。

## 議論
- **なぜRust/Wasmか？**: JS単体では数百・数千のSVG要素の座標計算やアニメーション制御がボトルネックになる可能性がある。Rustのメモリ安全性とWasmの実行速度を活かす。
- **なぜGoか？**: 型安全性、並行処理の強さ、およびVercelとの親和性が高いため。
- **Railsの不採用**: 今回の「大量のSVGを表示するアート作品」という用途では、サーバーサイドレンダリング中心のRailsよりも、クライアントサイドの計算能力（Wasm）と軽量なAPI（Go）の組み合わせが最適と判断した。

## 成果
- パフォーマンスの劇的な向上。
- モダンな言語（Go, Rust）による開発体験の向上。
- 既存のクラウドアイコン資産（SVG）を芸術的なアプローチで再利用。
